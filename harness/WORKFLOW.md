# Backend Agent Workflow

## 1. 태스크 문서 로드 (Task Load)

- `harness/tasks/task-NN-*.md` 파일을 읽어 작업의 **목적(Objective)**과 **제약(Out of Scope, Edge Cases 등)**을 파악한다.
- 태스크 문서는 "왜"와 "알려진 제약"만 담고 있다. "무엇을, 어떻게"는 다음 단계에서 코드를 읽어 도출한다.
- `CLAUDE.md`의 '문서 지도'를 참고하여 추가로 필요한 문서를 읽는다.

## 2. 코드베이스 탐색 (Codebase Exploration)

- 태스크 Objective를 기반으로 관련 파일을 능동적으로 읽는다.
- 탐색 목표:
  - 현재 구현의 어디에 문제가 있는가
  - 수정·생성이 필요한 파일은 무엇인가
  - 기존 코드 패턴에서 일관성 있게 적용할 수 있는 결정값은 무엇인가
  - 예상되는 사이드 이펙트와 엣지 케이스는 무엇인가

## 3. 계획 수립 및 승인 (Plan & Approval)

탐색 결과를 바탕으로 `PLAN.md`를 작성하고 사용자에게 보고한다.

**PLAN.md 포함 항목:**

- **작업 목표**: 태스크 Objective 요약 (재작성하지 않고 핵심만)
- **[도메인 수정 승인 요청]**: Domain Change가 [x]인 경우 최상단에 별도 섹션 작성
- **현황 분석**: 코드 탐색으로 파악한 현재 상태와 문제 위치
- **변경 파일 목록**: 수정·생성할 파일과 각 파일에서 할 일
- **구현 방향**: 핵심 결정사항 (기존 패턴에서 벗어나는 경우 그 이유 포함)
- **예상 사이드 이펙트**: 변경으로 인해 영향받는 다른 영역
- **테스트 전략**: 기본 프로토콜과 다른 경우에만 명시

명시적인 승인(Approve)이 떨어지기 전까지는 절대 코드를 작성하거나 수정하지 않는다.

## 4. 브랜치 생성 (Branching)

- 승인이 떨어지면 `main` 브랜치에서 분기하여 새로운 브랜치를 생성하고 이동한다.
- 브랜치 네이밍: `ai/[작업 카테고리]-[기능명]` (작업 카테고리는 feat, fix, refactor 등)
  - 기능명은 태스크 파일명을 기준으로 한다.
  - 예: `ai/feat-buzz-recipient-limit`

## 5. 구현 (Implementation)

- PLAN.md에서 승인된 방향대로 구현한다.
- 작업을 마친 뒤에는 작업한 범위의 테스트 코드를 작성 및 실행한다. 테스트 코드에 대한 규칙은 `TESTING-GUIDE.md`를 참고한다.
- [에러 통제]: 컴파일 에러나 예외 발생 시 스스로 원인을 분석하여 수정한다. 단, 동일한 에러로 3회 이상 해결에 실패할 경우, 코드 수정을 즉각 중단하고 사용자에게 상황을 보고한 뒤 지시를 대기한다.

## 6. 커밋 및 보고 (Commit)

- 작업이 완료되고 테스트가 통과하면, 변경 사항을 요약하여 커밋 메시지를 작성하고 스테이징 및 커밋한다.
