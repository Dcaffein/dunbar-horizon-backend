# Backend Agent Workflow

## 1. 태스크 문서 로드 (Task Load)
- 작업 시작 전 `harness/tasks/task-NN-*.md` 파일을 읽어 작업 범위를 파악한다.
- 태스크 문서의 각 섹션을 다음과 같이 활용한다.
  - `Objective` — PLAN.md의 작업 목표
  - `Domain Change` — [x] 있음이면 PLAN.md 상단에 `[도메인 수정 승인 요청]` 섹션 작성
  - `Target Files` — 수정·생성할 파일 목록
  - `Requirements` — 구현 요구사항
  - `Decisions` — 구현 중 판단이 필요한 정책값 (되묻지 않고 그대로 따른다)
  - `Testing Strategy` — 있으면 우선 적용, 없으면 `TESTING-PROTOCOL.md` 기본 규칙을 따른다
- `CLAUDE.md`의 '문서 지도'를 참고하여 추가로 필요한 문서를 읽는다.

## 2. 계획 수립 및 승인 (Plan & Approval)
- 태스크 문서를 기반으로 작업의 목표, 생성/수정할 파일 목록, 예상되는 사이드 이펙트를 포함하여 `PLAN.md`를 작성하고 사용자에게 보고한다.
- 명시적인 승인(Approve)이 떨어지기 전까지는 절대 코드를 작성하거나 수정하지 않는다.

## 3. 브랜치 생성 (Branching)
- 승인이 떨어지면 `main` 브랜치에서 분기하여 새로운 브랜치를 생성하고 이동한다.
- 브랜치 네이밍: `ai/[작업 카테고리]-[기능명]` (작업 카테고리는 feat, fix, refactor 등등) (기능명은 태스크 파일명을 기준으로 한다)
  - 예: `ai/feat-buzz-recipient-limit`

## 4. 구현 (Implementation)
- 작업을 마친 뒤에는 테스트 코드를 작성한다. 테스트 코드에 대한 규칙은 `TESTING-PROTOCOL.md`를 참고한다.
- [에러 통제]: 컴파일 에러나 예외 발생 시 스스로 원인을 분석하여 수정한다. 단, 동일한 에러로 3회 이상 해결에 실패할 경우, 코드 수정을 즉각 중단하고 사용자에게 상황을 보고한 뒤 지시를 대기한다.

## 5. 커밋 및 보고 (Commit)
- 작업이 완료되고 테스트가 통과하면, 변경 사항을 요약하여 커밋 메시지를 작성하고 스테이징 및 커밋한다.
