# Task Spec Guide

작업 명세서(Task Markdown)를 작성하는 가이드입니다.
이 가이드의 목표는 **작성자(사람)와 구현자(AI) 사이의 소통 비용을 줄이는 것**입니다.
명세서가 완결적일수록 PLAN.md 작성 전 되묻는 횟수가 줄어듭니다.

---

## 왜 이 가이드가 필요한가

명세서에서 소통 비용이 발생하는 주요 원인은 세 가지입니다.

1. **탐색 비용**: 대상 파일이 명시되지 않아 구현자가 직접 코드베이스를 뒤져야 함
2. **승인 불명확**: 도메인 계층 수정 여부가 모호해 PLAN.md 승인 단계에서 왔다 갔다 함
3. **정책값 미결정**: 구현에 필요한 구체적인 수치·전략·타입이 빠져 있어 구현 직전에 되물어야 함

---

## 섹션 구성

### 필수 섹션

```markdown
## Objective
(1–2문장) 이 작업이 해결하는 문제와 그 이유.
"무엇을 만드는가"가 아니라 "왜 필요한가"를 쓴다.
```

```markdown
## Domain Change
[ ] 없음 — domain/ 계층을 건드리지 않는다.
[x] 있음 — domain/ 파일을 수정한다. (PLAN.md에 별도 승인 섹션 필요)
```

> WORKFLOW.md 규칙: `domain/` 수정이 포함된 경우 PLAN.md 상단에 `[도메인 수정 승인 요청]` 섹션을 별도 작성해야 한다.
> 이 필드가 체크되어 있으면 구현자가 자동으로 처리한다.

```markdown
## Target Files
수정·생성할 파일의 경로와 역할을 나열한다.
경로가 확실하지 않아도 계층 수준(예: "account 도메인 서비스")까지만 써도 됨.

- `{domain}/domain/SomeEntity.java` — 검증 로직 추가
- `{domain}/adapter/in/web/dto/SomeRequest.java` — 입력 제약 어노테이션 추가
- `{domain}/application/service/SomeService.java` — 비즈니스 규칙 적용
```

```markdown
## Requirements
번호가 붙은 요구사항 목록. 각 항목은 단일 책임을 가진다.
"무엇을" 해야 하는지를 쓰고, "어떻게"는 구현자에게 위임한다.
단, 정책으로 결정한 구체적인 값(상수, 임계값, 타입 이름 등)은 여기에 명시한다.

1. ...
2. ...
```

```markdown
## Decisions (정책 결정값)
구현 전에 이미 결정된 정책값·전략·타입을 명시한다.
이 섹션이 비어 있거나 없으면 구현자가 PLAN.md 전에 되물어야 한다.

| 항목 | 결정값 | 비고 |
|------|--------|------|
| 상한값 예시 | 100 | 도메인 정책 기준 |
| 예외 타입 예시 | InvalidStateException | 기존 예외 재사용 |
```

### 선택 섹션

아래 섹션은 해당되는 경우에만 추가한다. 없으면 구현자는 기본 프로토콜을 따른다.

```markdown
## Out of Scope
이 작업에서 명시적으로 제외하는 항목.
"하면 안 된다"가 아니라 "나중에 별도 작업으로 한다"는 의미로 쓴다.

- 인프라 레벨의 설정(예: 정책, 스케줄러)은 애플리케이션 코드에서 제외
```

```markdown
## Edge Cases
구현 중 특별히 처리해야 하는 경계 상황.
알려진 것만 쓴다. 모르면 생략해도 된다 — 구현자가 발견하면 PLAN.md에 기술한다.

- 요청자가 대상 리소스의 소유자인 동시에 다른 역할을 가지는 경우
- 입력값이 null 또는 빈 컬렉션인 경우
```

```markdown
## Testing Strategy
단위 테스트와 통합 테스트 중 무엇이 필요한지, 어느 수준까지 커버할지를 지정.
기본값: TESTING-PROTOCOL.md 규칙을 따른다. 기본값과 다른 경우에만 이 섹션을 추가한다.

- Unit: 외부 어댑터를 Mock하여 서비스 레이어 로직만 검증
- Integration: Testcontainers를 활용하여 실제 인프라와의 연동 검증
```

```markdown
## API Contract Change
[ ] 없음
[x] 있음

변경 전: METHOD /api/v1/{resource} — 기존 요청/응답 스펙
변경 후: METHOD /api/v1/{resource} — 변경된 요청/응답 스펙
```

> 이 필드가 체크되어 있으면 구현자가 컨트롤러·DTO까지 함께 수정한다.

```markdown
## Dependencies
이 작업이 선행되어야 하거나, 이 작업 이후에 연결되는 태스크.

- 선행: task-NN (작업명) 완료 후 진행
- 후행: task-NN (작업명)에서 이 결과를 사용
```

---

## 완결성 체크리스트

명세서를 제출하기 전 아래를 확인한다.

- [ ] Objective가 "왜"를 설명하는가 (문제의 원인)
- [ ] Domain Change 여부가 체크되어 있는가
- [ ] Target Files에 계층 수준 이상의 정보가 있는가
- [ ] 구현에 필요한 상수·임계값·타입 이름이 Decisions에 있는가
- [ ] API 변경이 있다면 변경 전/후 스펙이 적혀 있는가
